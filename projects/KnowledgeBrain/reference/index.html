    怎么理解“召回”¶

    在信息检索和机器学习领域，“召回”是一个重要的评价指标，用于衡量模型或系统在检索或分类任务中的表现。具体来说，召回率（Recall）表示的是模型从所有相关样本中正确识别出相关样本的比例。它的计算公式是：

    \[ \text{召回率} = \frac{\text{正确识别的相关样本数}}{\text{所有相关样本数}} \]

    举个例子，如果有100个相关文档，而模型正确检索出了80个相关文档，那么召回率就是80%。

    召回率的高低反映了模型在识别相关样本方面的能力。高召回率意味着模型能够识别出大多数的相关样本，但可能会带来更多的误报（即非相关样本被错误地识别为相关样本）。因此，召回率通常与精确率（Precision）一起使用，以全面评估模型的性能。

    在实际应用中，召回率的重要性取决于具体的任务需求。例如，在医疗诊断中，高召回率可能更为重要，因为漏诊可能带来严重后果。

    向量召回是什么¶

    向量召回是一种基于向量空间模型的检索方法。它通过将文档和查询表示为向量，并计算它们之间的相似度来进行检索。常用的相似度度量方法包括余弦相似度、欧几里得距离等。

    具体步骤如下：

    1. 向量化：将文档和查询转换为向量表示。
    2. 计算相似度：计算查询向量与文档向量之间的相似度。
    3. 排序：根据相似度对文档进行排序，返回最相关的文档。

    这种方法在处理大规模文本数据时非常有效，尤其是在自然语言处理和信息检索领域。

    使用FAISS进行向量召回的步骤¶

    FAISS（Facebook AI Similarity Search）是一个高效的相似性搜索库，特别适用于大规模向量检索。它可以快速地进行向量索引和查找，适合处理高维向量数据。

    什么是bm25召回¶

    BM25（Best Matching 25）是一种基于概率检索模型的文档检索算法，用于计算查询与文档之间的相关性得分。它是Okapi BM25的简化版本，广泛应用于信息检索系统中。

    BM25召回的步骤如下：

    1. 文档预处理：对文档进行分词、去停用词等预处理操作。
    2. 计算词频：计算每个词在文档中的词频（Term Frequency, TF）。
    3. 计算逆文档频率：计算每个词的逆文档频率（Inverse Document Frequency, IDF）。
    4. 计算BM25得分：根据BM25公式计算查询与每个文档的相关性得分。

    什么是RAG¶

    RAG（Retrieval-Augmented Generation）是一种结合检索和生成的自然语言处理技术。它通过从一个大型文档集合中检索相关信息，然后使用生成模型来生成最终的回答或文本。RAG 模型通常用于问答系统、对话系统和其他需要结合外部知识库的任务。

    RAG 的工作流程如下：

    1. 检索（Retrieval）：首先，从一个大型文档集合（如知识库或互联网）中检索与输入查询相关的文档或段落。常用的检索方法包括 BM25、FAISS 等。
    2. 生成（Generation）：然后，使用生成模型（如 GPT-3、BERT 等）结合检索到的文档生成最终的回答或文本。

    这种方法的优点是能够利用外部知识库中的丰富信息，提高生成文本的准确性和相关性。

    下面是一个简单的 RAG 工作流程的伪代码：

    1. 输入查询 query
2. 使用检索模型从文档集合中检索相关文档 documents
3. 将 query 和 documents 作为输入，传递给生成模型
4. 生成模型生成最终的回答或文本 response
5. 输出 response

    RAG 模型的实现可以使用现有的检索和生成工具，如 FAISS 进行检索，GPT-3 进行生成。

    什么是 重排序¶

    Reranker 是信息检索生态系统中的一个重要组成部分，用于评估搜索结果，并进行重新排序，从而提升查询结果相关性。在 RAG 应用中，主要在拿到召回结果后使用 Reranker，能够更有效地确定文档和查询之间的语义相关性，更精细地对结果重排，最终提高搜索质量。 将 Reranker 整合到 RAG 应用中可以显著提高生成答案的精确度，因为 Reranker 能够在单路或多路的召回结果中挑选出和问题最接近的文档。此外，扩大检索结果的丰富度（例如多路召回）配合精细化筛选最相关结果（Reranker）还能进一步提升最终结果质量。使用 Reranker 可以排除掉第一层召回中和问题关系不大的内容，将输入给大模型的上下文范围进一步缩小到最相关的一小部分文档中。通过缩短上下文， LLM 能够更“关注”上下文中的所有内容，避免忽略重点内容，还能节省推理成本。 向量召回中使用的是bi-encoder结构，而bge-reranker-large 使用的是 cross-encoder结构，cross-encoder结构一定程度上要优于bi-encoder 可以看出，这个检索系统包含两个阶段： 在向量数据库中检索出 Top-K 相关文档，同时也可以配合 Sparse embedding（稀疏向量模型，例如TF-DF）覆盖全文检索能力。 Reranker 根据这些检索出来的文档与查询的相关性进行打分和重排。重排后挑选最靠前的结果作为 Prompt 中的Context 传入 LLM，最终生成质量更高、相关性更强的答案。 但是需要注意，相比于只进行向量检索的基础架构的 RAG，增加 Reranker 也会带来一些挑战，增加使用成本。 这个成本包括两方面，增加延迟对于业务的影响、增加计算量对服务成本的增加。我们建议根据自己的业务需求，在检索质量、搜索延迟、使用成本之间进行权衡，合理评估是否需要使用 Reranker。 重排序在搜索引擎、问答系统和推荐系统中广泛应用，通过结合初始检索和重排序模型，可以显著提高检索结果的质量和用户满意度。
